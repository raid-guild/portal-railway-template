# Prism / Portal Memory Handoff

Status: planning handoff
Last updated: 2026-06-02

## Purpose

This handoff is for a follow-up implementation session in the Prism repo. The
goal is to make Prism Memory and Portal reinforce each other without tightly
coupling the systems or creating a feedback loop.

Portal is becoming the canonical presentation and coordination layer for:

- projects
- threads
- sessions/events
- activity items
- profiles
- daily briefs

Prism should remain the memory, artifact, retrieval, and synthesis layer.

## Current Problem

Prism Memory currently has project-state behavior, but the project detection is
too coupled to Discord channel structure. The default shape is effectively:

- look at raw Discord bucket records
- detect projects from configured buckets
- infer project keys from channel prefixes such as `raid-`

That is too brittle for the current Portal direction. Useful project and session
context is increasingly represented in Portal, not only in Discord channel names.

The failure mode is:

- real work happens in meetings, Discord, sessions, and Portal
- Prism captures memory and summaries
- Prism project state stays empty or inaccurate because the activity does not
  match project-channel detection rules
- Portal daily briefs become isolated summaries instead of linked updates to
  activity, projects, and threads

## Desired Boundary

Use this separation:

```txt
Portal = canonical coordination and presentation records
Prism = memory, raw artifacts, retrieval, and synthesis
Agent workflow = mapping layer between them
```

Prism should not directly own Portal records. Portal should not try to become the
memory engine.

## Recommended Data Flow

```txt
Portal context export
  -> Prism context/knowledge ingest
  -> Prism Discord + meeting collection
  -> Prism memory/digest/artifact synthesis
  -> Portal update packet candidates
  -> Portal agent workflow resolves/upserts records
  -> Portal daily brief links activity, threads, projects, events
```

The key change is that Prism should pull Portal context before or during
synthesis so it can resolve mentions against known projects, sessions, threads,
and profiles.

## Portal Context Prism Should Pull

Prism should ingest a read-only Portal context snapshot. Start with public,
authenticated, and member-visible records only when the API token has the proper
scope.

Recommended objects:

### Projects

- id
- slug
- title
- summary
- projectStatus
- visibility
- lastActiveAt
- stewards
- contributors
- relatedProjects
- threads
- events
- links
- sourceEvidence

### Threads

- id
- slug
- title
- summary
- threadStatus
- lastActiveAt
- participants
- relatedProjects
- links
- visibility

### Events / Sessions

- id
- title
- summary
- sessionType
- startsAt
- endsAt
- visibility
- hostProfiles
- speakerProfiles
- relatedProjects
- relatedThreads
- relatedProfiles
- discordScheduledEventID
- discordEventURL
- joinURL
- artifacts/resources if exposed

Use both future and recent past sessions:

- future sessions help Prism understand expected context
- past sessions help Prism connect meeting summaries and artifacts to Portal

### Activity Items

- id
- title
- body
- activityType
- happenedAt
- sourceLabel
- sourceURL
- relatedProject
- relatedThread
- relatedEvent

These are useful as recent factual history, but should be treated carefully if
they were agent-generated from Prism.

### Profiles

Only import basic identity fields needed for attribution:

- id
- displayName
- handle
- visibility
- linked user/email only if needed and allowed

## Avoiding Feedback Loops

The biggest risk is Prism consuming Portal records that were generated by Prism,
then re-summarizing them as new facts.

Use these loop-prevention rules:

1. Treat raw Discord messages, meeting transcripts, meeting summaries, and human
   Portal edits as stronger evidence than generated briefs.
2. Import Portal records into a separate namespace such as `portal-context`, not
   the same lane as raw memory artifacts.
3. Mark imported Portal records with source metadata:
   - `sourceSystem: portal`
   - `sourceCollection`
   - `sourceID`
   - `updatedAt`
   - `_status`
   - `originSystem` when known
4. Do not use daily briefs as primary evidence for the next daily brief.
5. Prefer Portal projects, threads, sessions, and human/admin activity items as
   context.
6. Treat agent-created Portal activity as derived unless it has been reviewed or
   edited by a human.
7. Generate stable update keys for candidates so Portal can upsert idempotently.

Prism agent workflow produce a small update packet.

## Prism Improvements To Consider

### 1. Add Portal Context Ingest

Add a read-only Portal context fetch step. This could be either:

- a new Prism source adapter
- a memory/knowledge source sync
- a daily context import before memory synthesis

Start with one endpoint if Portal exposes it later:

```txt
GET /api/portal/context
```

Fallback to existing Payload collection APIs:

```txt
GET /api/projects
GET /api/threads
GET /api/events
GET /api/activityItems
GET /api/profiles
```

### 2. Use Portal Context For Entity Resolution

When Prism sees "sessions", "Portal", "onboarding", "Demo Day", or a member
handle, it should resolve those mentions against Portal context before inventing
new project/thread labels.

### 3. Include Meeting Summaries In Project/Thread State

Current project-state detection appears focused on raw Discord bucket records.
Meeting summaries and processed inbox memory should also feed project/thread
state because they are often higher-signal than raw chat.

### 4. Move From Channel-Derived Projects To Canonical Project Aliases

Project detection should prefer:

1. explicit Portal project IDs/slugs from event/session metadata
2. Portal project aliases/source evidence
3. known configured project aliases in Prism state
4. channel prefix fallback as a last resort

Channel structure should be supporting evidence, not the project model.

### 5. Add Thread Candidate Extraction

Prism already has Discord thread promotion for active Discord threads. Portal
threads are different: persistent lines of work or thought.

Add extraction for Portal-style threads:

- recurring topic
- ongoing open question
- workstream discussed across multiple sessions
- topic tied to multiple activity items

Prefer updating existing Portal threads over creating new ones.

### 6. Output Reviewable Candidates

Keep default behavior review-first. Prism can generate candidates confidently,
but Portal should decide whether records are draft or published.

## Implementation Sequence

Recommended first pass:

1. Add a Prism Portal context fetcher that reads Portal projects, threads,
   events, recent activity, and profiles.
2. Store fetched context separately from raw memory, for example:
   `state/current/portal-context.json` or a `knowledge/sources/portal` partition.
3. Update meeting/digest synthesis prompts or code to use Portal context for
   entity resolution.
4. Add a Portal update packet artifact generated by the daily memory job.
5. Keep the packet as an artifact first, without writing to Portal.
6. Once packet quality is good, wire the Portal agent workflow to consume it and
   upsert records.

## Open Questions

- Should Portal expose a purpose-built `/api/portal/context` endpoint, or should
  Prism read collection APIs directly?
- Should Portal context live in Prism memory, Prism knowledge, or separate state?
- How should Portal mark records as human-reviewed vs agent-derived?
- What stable external key format should Portal use for Prism-generated
  candidates?
- Should daily brief generation live in Prism, Portal, or the agent workflow?

## Working Assumption

The cleanest near-term design is:

- Prism pulls Portal context read-only.
- Prism emits a reviewable Portal update packet.
- Portal agent workflow consumes the packet and performs idempotent writes.
- Portal daily briefs link the created/updated activity, threads, projects, and
  sessions.

This gives Prism enough canonical structure to understand community activity
without making Portal and Prism recursively summarize each other.
